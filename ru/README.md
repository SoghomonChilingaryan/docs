# Skeleton: правила и шаблоны

Это единый источник правды о том, как устроено приложение и как добавлять новый код без нарушения архитектуры.

## Как читать

Рекомендуемый порядок:

1. `architecture.md`: правила слоев, зависимости, DI, навигация.
2. `feature_template.md`: как заводить новую фичу и не ломать границы модулей.
3. `rebrand_theme_phase1.md`: если есть задача безопасного rebrand темы.

## Содержание

- `architecture.md`: правила Clean Architecture, слои, границы модулей.
- `feature_template.md`: шаблон фичи (папки, нейминг, ответственность).
- `rebrand_theme_phase1.md`: стратегия миграции темы (legacy + rebrand параллельно).

## Цели

- Сделать кодовую базу предсказуемой между проектами.
- Предотвращать нарушения слоев (presentation -> domain -> data).
- Сделать rebrand безопасным: менять UI без риска сломать бизнес-логику.

## Базовые модули (то, что есть до фич)

Текущее ядро проекта (не бизнес-фичи):

- `lib/src/common/core/app`: bootstrap приложения (запуск, склейка окружения/DI).
- `lib/src/common/core/config`: `AppConfig` и адаптеры конфигурации.
- `lib/src/common/core/di`: DI контейнер и сборка dependency graph.
- `lib/src/common/theme`: тема приложения (legacy/rebrand по мере роста).
- `lib/src/navigation`: контракт навигации и адаптеры.

Бизнес-фичи живут в `lib/src/features/<feature_name>` и заводятся по шаблону.

## Папка packages (рекомендуемо)

Если в проекте есть большие внешние интеграции (серверный API, DB/кэш для больших данных, тяжелые SDK), рекомендуется выносить реализацию в `packages/`, а в приложении держать только контракты в `lib/src/common/domain` и связывать через DI.

## Про docs как стандарт

Этот набор документов задуман как “общий стандарт” и может жить отдельно от skeleton-репозитория (например, как отдельный репозиторий/подмодуль), чтобы переиспользоваться во всех приложениях.

## Как мы запускаем приложение (кратко)

- сначала Splash собирает DI (асинхронно)
- после DI создаем глобальные блоки (например, Settings для theme/locale)
- затем строим `MaterialApp`
